# librefang-cli-templates

Static TOML configuration templates used by `librefang init` to scaffold a new LibreFang Agent OS configuration file.

## Overview

This module contains no executable code. It ships two TOML template files that are rendered at project initialization time. Placeholders in the form `{{key}}` are substituted with user-provided values collected during the init or setup-wizard flow.

| File | Purpose |
|---|---|
| `init_default_config.toml` | Full reference config with every section present (most commented out). Used by `librefang init` when the user wants a complete starting point. |
| `init_wizard_config.toml` | Minimal skeleton generated by the interactive setup wizard (`librefang init --wizard`). Only essential keys are emitted. |

## Template Rendering

Both files are plain TOML with Mustache-style placeholders. The CLI init command reads the appropriate file, performs string substitution, and writes the result to the user's config directory.

### Placeholders

| Placeholder | Used In | Description |
|---|---|---|
| `{{provider}}` | Both | LLM provider name (e.g. `openai`, `anthropic`, `ollama`) |
| `{{model}}` | Both | Default model identifier (e.g. `gpt-4o`, `claude-sonnet-4-20250514`) |
| `{{api_key_env}}` | `init_default_config.toml` | Environment variable name holding the API key (e.g. `OPENAI_API_KEY`) |
| `{{api_key_line}}` | `init_wizard_config.toml` | Pre-formatted `api_key_env = "..."` line (or empty string if not applicable) |
| `{{routing_section}}` | `init_wizard_config.toml` | Optional agent routing TOML block appended after memory config |

No template engine dependency is required — simple string replacement (`str.replace`) suffices.

## Configuration Sections Reference

The sections below appear in `init_default_config.toml` as inline documentation. When modifying the template, keep comments accurate and in sync with the runtime config parser.

### Core Settings (top-level keys)

| Key | Default | Values |
|---|---|---|
| `api_listen` | `"127.0.0.1:4545"` | Bind address and port. Loopback-only by default. |
| `log_level` | `"info"` | `trace` \| `debug` \| `info` \| `warn` \| `error` |
| `mode` | `"default"` | `stable` \| `default` \| `dev` |
| `update_channel` | `"stable"` | `stable` \| `beta` \| `rc` |

### Security Model

The template enforces a safe-by-default posture:

- **Non-loopback binding is rejected** unless authentication is configured. The daemon checks for `api_key`, `dashboard_user`/`dashboard_pass`, or `[[users]]` entries before accepting a non-loopback `api_listen`.
- `LIBREFANG_ALLOW_NO_AUTH=1` overrides this guard (documented as not recommended).
- Dashboard defaults (`librefang` / `librefang`) must be changed after first login.
- Secure credential storage is supported via vault references (`"vault:dashboard_password"`) or environment variables (`LIBREFANG_DASHBOARD_PASS`).

### Terminal Access Control (`[terminal]`)

Commented out by default. Three keys control remote shell access:

- `allow_remote` — enables remote terminal connections (requires auth unless overridden).
- `allow_unauthenticated_remote` — foot-gun guard; must be explicitly set to `true`.
- `require_proxy_headers` — enable only behind a reverse proxy that sets `X-Forwarded-For` / `X-Real-IP`.

### LLM Configuration (`[default_model]`)

The primary model used when no agent-specific override exists.

```toml
[default_model]
provider = "{{provider}}"
model = "{{model}}"
api_key_env = "{{api_key_env}}"
# base_url = ""    # optional endpoint override
```

### Memory (`[memory]` and `[proactive_memory]`)

- `[memory].decay_rate` — confidence decay per cycle (default `0.05`).
- `[proactive_memory]` controls automatic fact extraction and recall. Enabled by default with `auto_memorize = true` and `auto_retrieve = true`.

### Web Tools (`[web]`)

- `search_provider = "auto"` auto-detects among Tavily → Brave → Jina → Perplexity → DuckDuckGo based on available API keys.
- `[web.fetch]` controls page extraction limits and SSRF protection. Cloud metadata addresses (`169.254.x.x`, `100.64.x.x`) are always blocked.

### Task Queue (`[queue.concurrency]`)

Concurrency caps per lane:

| Lane | Default | Purpose |
|---|---|---|
| `main_lane` | 3 | User messages |
| `cron_lane` | 2 | Scheduled jobs |
| `subagent_lane` | 3 | Child agents |
| `trigger_lane` | 8 | In-flight trigger dispatches |
| `default_per_agent` | 1 | Per-agent fallback |

### Shell Execution Policy (`[exec_policy]`)

```toml
[exec_policy]
mode = "deny"           # deny | allowlist | full
timeout_secs = 30
max_output_bytes = 102400
```

`deny` is the safest default — no shell commands execute until the operator explicitly opens access.

### Hot-Reload (`[reload]`)

```toml
[reload]
mode = "hybrid"     # off | restart | hot | hybrid
debounce_ms = 500
```

`hybrid` applies hot-reload where possible and restarts the daemon for keys that require it.

### Optional Sections (commented out)

These sections are present in `init_default_config.toml` but commented out. Uncomment to activate:

| Section | Purpose |
|---|---|
| `[[fallback_providers]]` | LLM failover chain when the default provider fails |
| `[rate_limit]` | GCRA-based per-IP rate limiting for API and WebSocket |
| `[registry]` | Agent registry sync interval |
| `[compaction]` | LLM-based session history summarization |
| `[triggers]` | Event trigger recursion and cooldown limits |
| `[budget]` | Per-hour/day/month cost caps with per-provider overrides |
| `[thinking]` | Extended thinking budget for Claude models |
| `[channels.telegram/discord/slack/wechat]` | Messaging channel integrations |
| `[[mcp_servers]]` | External MCP tool servers |
| `[browser]` | Headless browser automation |
| `[docker]` | Containerized code execution sandbox |
| `[inbox]` | File-based async message ingestion |
| `[network]` | P2P federation between LibreFang nodes |
| `[provider_regions]` | Regional endpoint selection (e.g., Qwen intl, MiniMax China) |
| `[provider_urls]` | Custom provider endpoints (Ollama, vLLM, etc.) |

### Editor Integration (ACP)

Documented in the template comments but requires no configuration keys. Editors spawn `librefang acp` as a child process. The ACP server auto-selects between in-process mode and daemon-attached mode (via `~/.librefang/acp.sock` on Unix).

## Relationship to the Codebase

```
┌──────────────────────┐
│  librefang init      │
│  (CLI command)       │
└──────┬───────────────┘
       │ reads template + collects user input
       ▼
┌──────────────────────────────────┐
│  templates/                      │
│    init_default_config.toml      │──── string replace ────▶  ~/.librefang/config.toml
│    init_wizard_config.toml       │
└──────────────────────────────────┘
                                                    │
                                                    ▼
                                          ┌──────────────────┐
                                          │  config loader    │
                                          │  (runtime)        │
                                          └──────────────────┘
```

The templates are a one-time scaffolding concern. After generation, the user edits `~/.librefang/config.toml` directly and the runtime config loader parses it. The templates have no further involvement at runtime.

## Modifying Templates

When adding a new configuration key:

1. Add the key to `init_default_config.toml` in the appropriate section, commented out with a descriptive comment.
2. If the key is essential for first run, add it (with placeholder if needed) to `init_wizard_config.toml`.
3. Ensure the init command's substitution logic handles any new `{{placeholder}}` tokens.
4. Keep inline comments concise — they serve as in-file documentation for operators who never read external docs.
