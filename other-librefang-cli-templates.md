# librefang-cli-templates

Configuration templates used during LibreFang Agent OS initialization. This module contains two TOML template files that are rendered with user-provided values and written to disk as the initial `config.toml`.

## Templates

### `init_default_config.toml`

The **full reference template** — every configuration section the platform supports, with sensible defaults for a local deployment. Most optional sections are commented out so the file doubles as self-documenting reference material.

Populated by `librefang init` (non-interactive mode) or when the user explicitly requests a full config.

**Placeholders:**

| Placeholder | Description | Example rendered value |
|---|---|---|
| `{{provider}}` | Default LLM provider identifier | `"openai"` |
| `{{model}}` | Default model name | `"gpt-4o"` |
| `{{api_key_env}}` | Environment variable holding the API key | `"OPENAI_API_KEY"` |

### `init_wizard_config.toml`

The **minimal template** — only the sections strictly required to start the agent. Generated by `librefang init --wizard` (interactive setup) after the user answers a few prompts.

**Placeholders:**

| Placeholder | Description | Example rendered value |
|---|---|---|
| `{{provider}}` | Default LLM provider identifier | `"openai"` |
| `{{model}}` | Default model name | `"gpt-4o"` |
| `{{api_key_line}}` | Complete `api_key_env` line (or empty string) | `api_key_env = "OPENAI_API_KEY"` |
| `{{routing_section}}` | Optional routing configuration block | Multi-line TOML or empty string |

The wizard template uses coarser placeholders (`{{api_key_line}}`, `{{routing_section}}`) because the interactive flow may produce multi-line blocks or omit sections entirely depending on user choices.

## Configuration Sections

The full template organizes settings into these groups:

### Core Server

| Key | Default | Purpose |
|---|---|---|
| `api_listen` | `127.0.0.1:4545` | Bind address. Use `0.0.0.0:4545` for LAN access. |
| `log_level` | `info` | Verbosity: `trace`, `debug`, `info`, `warn`, `error` |
| `mode` | `default` | Runtime mode: `stable`, `default`, `dev` |
| `update_channel` | `stable` | Release track: `stable`, `beta`, `rc` |

### Authentication

`dashboard_user` / `dashboard_pass` — credentials for the web dashboard. The template includes inline comments pointing to secure alternatives (Vault integration, environment variable `LIBREFANG_DASHBOARD_PASS`).

### Default Model (`[default_model]`)

Minimum required: `provider`, `model`, and `api_key_env`. An optional `base_url` override supports local proxies or self-hosted endpoints.

### Memory System

- **`[memory]`** — Base decay rate for confidence scoring.
- **`[proactive_memory]`** — Auto-extraction and auto-retrieval of conversational facts. Enabled by default with tunable thresholds and limits.

### Web Tools (`[web]`)

Search provider auto-detection cascades through Tavily → Brave → Jina → Perplexity → DuckDuckGo based on available API keys. The `[web.fetch]` sub-section controls page extraction limits, timeouts, and SSRF protection.

### Task Queue (`[queue.concurrency]`)

Separate concurrency lanes for user messages (`main_lane`), scheduled jobs (`cron_lane`), and child agents (`subagent_lane`).

### Shell Execution (`[exec_policy]`)

Three modes control command execution:
- **`deny`** — No shell access (default).
- **`allowlist`** — Only pre-approved commands.
- **`full`** — Unrestricted (use with caution).

### Hot-Reload (`[reload]`)

`mode` controls how config changes take effect:
- **`off`** — Ignore changes until manual restart.
- **`restart`** — Restart the process on change.
- **`hot`** — Apply in-place without restart.
- **`hybrid`** — Hot-apply safe changes, restart for structural ones.

Debouncing prevents rapid re-application during bulk edits (`debounce_ms`).

### Optional Sections (commented out)

These are present but disabled in the full template. Uncomment to activate:

| Section | Purpose |
|---|---|
| `[terminal]` | Remote terminal access control and proxy header trust |
| `[provider_regions]` | Regional endpoints for providers like Qwen, MiniMax |
| `[provider_urls]` | Override provider API URLs (e.g., local Ollama, vLLM) |
| `[[fallback_providers]]` | LLM failover chain when primary is unavailable |
| `[rate_limit]` | GCRA-based API and WebSocket rate limiting |
| `[registry]` | Agent registry cache TTL |
| `[compaction]` | LLM-based session history summarization thresholds |
| `[triggers]` | Event trigger cooldown, depth, and timeout limits |
| `[budget]` | Hourly/daily/monthly cost caps with per-provider overrides |
| `[thinking]` | Extended thinking budget for Claude models |
| `[channels.*]` | Messaging integrations: Telegram, Discord, Slack, WeChat |
| `[[mcp_servers]]` | Model Context Protocol external tool servers |
| `[browser]` | Headless browser automation settings |
| `[docker]` | Docker sandbox for isolated code execution |
| `[inbox]` | File-based async message ingestion |
| `[network]` | P2P federation between LibreFang nodes |

## How Templates Are Rendered

```mermaid
flowchart LR
    A[User runs<br>librefang init] --> B{Interactive?}
    B -- --wizard --> C[init_wizard_config.toml]
    B -- default --> D[init_default_config.toml]
    C --> E[Replace placeholders]
    D --> E
    E --> F[Write config.toml]
```

The CLI's init command selects a template, substitutes `{{placeholder}}` values collected from CLI flags or interactive prompts, and writes the result. Because the templates are pure TOML with Mustache-style placeholders, no template engine dependency is required — simple string replacement suffices.

## Adding New Configuration

When adding a new configuration section to the platform:

1. Add it (commented out) to `init_default_config.toml` with a descriptive header comment and sensible defaults.
2. If the section is commonly needed during first-time setup, add a minimal version to `init_wizard_config.toml` with any necessary placeholders.
3. Include inline comments explaining each key — the full template is many users' primary reference for what's available.
